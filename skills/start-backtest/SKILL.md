# 启动回测

🚨 **重要提醒：回测提交成功后绝对不要询问用户是否需要等待或查看结果！** 🚨

🚨 **强制规则：每次回测必须询问时间范围！绝对不要使用历史对话中的时间信息！** 🚨

🚨 **强制规则：调用Python脚本前必须获取并传递token参数！** 🚨

## ⚠️ 重要：策略组回测 = 多策略回测

**策略组回测规则：**
- 策略组回测本质上就是多策略回测
- 使用完全相同的Python接口和参数  
- 需要先获取策略组内所有策略的ID，然后调用 `--strategy-ids`
- 保证金分配规则与多策略完全相同
- **共享模式支持任意比例分配**（如100,100表示各自10000保证金）

⚠️ **重要规则：每次回测都是新的独立任务**
- **不要查询**之前的回测记录
- **不要读取**历史回测数据 
- **即使策略相同**，也要重新开启新回测
- 每次回测的参数（保证金模式、时间范围等）都可能不同

⏰ **回测对话单元独立性规则：**
- **✅ 可以读取聊天记录，了解历史发生了什么**
- **🚫 但每次回测都是独立的对话单元**
- **不能跨对话单元复用回测参数**（时间范围、保证金模式等）
- **✅ 每次回测都必须在当前对话单元中完整询问所有必要参数**
- 除非用户在当前请求中已明确指定参数

启动新的回测任务，支持查看策略组列表、策略列表，以及开始回测。

## 🔑 Token 获取和传递规则（重要）

### ⚠️ Python脚本调用前的必要步骤

**问题**：Python脚本有时无法自动识别当前工作区，导致token获取失败

**解决方案**：Agent在调用任何Python脚本前，必须按以下步骤操作：

**步骤1：获取当前机器人agentID**
```bash
# Agent应该知道自己的agentID，直接使用
# 例如：qc-4b04623f0831
```

**步骤2：从users.json获取对应token**
```bash
# 根据当前机器人的agentID查找对应的token
cat ~/.quantclaw/users.json | jq -r --arg agent_id "当前机器人的agentID" '.users[] | select(.agentId == $agent_id) | .token'
```

**步骤3：传递token给Python脚本**
```bash
# 必须使用--token参数传递，不要依赖脚本自动获取
TOKEN="获取到的token"
python skills/start-backtest/start.py --token "$TOKEN" [其他参数]
```

### 🚨 强制执行格式

**❌ 错误做法**：
```bash
# 不要直接调用，可能无法获取token
python skills/start-backtest/start.py --list-strategies
```

**✅ 正确做法**：
```bash
# Agent使用自己的agentID获取token，再传递给脚本
TOKEN=$(cat ~/.quantclaw/users.json | jq -r --arg agent_id "当前机器人的agentID" '.users[] | select(.agentId == $agent_id) | .token')
python skills/start-backtest/start.py --token "$TOKEN" --list-strategies
```

### 📋 适用的所有Python脚本调用

**必须使用token传递的命令**：
```bash
# 查看策略组列表
python skills/start-backtest/start.py --token "$TOKEN" --list-groups

# 查看策略列表  
python skills/start-backtest/start.py --token "$TOKEN" --list-strategies

# 查看回测详情
python skills/start-backtest/start.py --token "$TOKEN" --detail 5906

# 启动回测
python skills/start-backtest/start.py --token "$TOKEN" --apply --strategy-ids 4300,4679 --bgn-date 2024-01-01 --end-date 2024-12-31 --leverage 10
```

### ⚠️ 重要提醒

- **每次调用Python脚本都必须先获取token**
- **不要依赖脚本的自动token获取功能**
- **Agent应该知道自己的agentID**（如qc-4b04623f0831）
- **使用自己的agentID从users.json获取token**
- **token获取失败时应该报错并停止执行**

### 💡 AgentID说明
- **Agent自己知道自己的身份标识**
- **不需要从文件路径解析agentID**
- **直接使用已知的agentID查找token**

## 🔄 回测模式识别（最重要）

### 📋 单策略回测（不需要询问保证金模式）

**⚠️ 重要区别：真正的单策略 vs 多个单独策略**

**真正的单策略回测触发条件：**
- 用户明确指定**一个**策略进行回测
- 用户说"回测策略XXX"（指定具体策略ID/名称）
- 用户说"用这个策略回测"（上下文中只有一个策略）
- 用户选择了一个策略后说"回测"

**多个独立策略回测触发关键词：**
- "分别回测"（多个策略，分别独立执行）
- "单独回测"（多个策略，每个单独执行）
- "独立回测"（多个策略，各自独立执行）
- "单独测试"、"分开回测"、"各自回测"

**处理方式：**
- **真正单策略**：1个策略 → 1个回测任务
- **多个独立策略**：N个策略 → N个独立回测任务
- **都不需要询问保证金模式**（单策略默认独占）
- **🚫 但必须询问回测时间范围**
- 确认时间后执行对应数量的独立回测任务

**标准流程：**
```
用户："分别回测我最新的2个风霆V4.2的SOL策略"

Step 1: 查询策略列表，找到最新2个SOL策略
Step 2: 显示找到的策略
Step 3: 询问："请问回测的时间范围是？例如：最近7天、最近30天、最近3个月、最近半年，或具体日期范围"
Step 4: 用户回复时间范围后，执行回测
```

### ⚠️ 多策略/策略组回测强制流程（需要询问保证金模式）

**当检测到以下情况时，必须先询问用户，不能直接执行回测：**
- "一起回测"、"同时回测"、"组合回测"
- 用户提到"策略组"、"策略组回测"
- 明确要求多个策略在同一个回测中运行
- 策略组名称/ID + "回测"的组合

### 强制执行步骤：

**步骤1：正确识别回测类型**

**如果用户说"分别回测"、"单独回测"、"独立回测"：**
- 这是多个**单策略回测**
- **不要询问保证金模式**
- **但必须询问回测时间范围**（除非用户已明确指定）
- 确认时间后执行相应数量的独立回测任务

**如果用户说"一起回测"、"同时回测"、"组合回测"：**
- 这是**多策略回测**  
- **必须询问保证金模式**：

```
您要将多个策略一起回测，请选择保证金模式：

1️⃣ **独占模式** - 每个策略独立使用保证金，互不影响
2️⃣ **共享模式** - 多个策略共享同一保证金池

请回复 1 或 2
```

**步骤2：等待用户回复**

- 用户回复 1 → 使用独占模式，继续执行回测
- 用户回复 2 → 进入步骤3

**步骤3：如果用户选择共享模式，询问保证金分配比例**

```
您选择了共享模式，请为每个策略分配保证金比例：
- 策略A (xxx)：？%
- 策略B (xxx)：？%
- 策略C (xxx)：？%

💡 **分配规则：**
✅ 每个比例是相对于基础保证金(10000)的百分比
✅ 100,100 表示两个策略各分配10000保证金
✅ 支持任意比例配置,但是单个策略分配到的保证金比例不能大于100，如50,100表示5000和10000
✅ 按策略列表顺序对应分配

格式：用逗号分隔，如：60,40 或 100,100 或 50,100,100
```

**步骤4：策略组回测专用流程**

**如果是策略组回测，按以下步骤询问：**

1. **策略组选择确认**：
   ```
   检测到策略组回测需求，请确认要回测的策略组：
   - 策略组名称/ID：XXX
   - 是否正确？请回复"是"继续，或提供正确的策略组信息
   ```

2. **保证金模式选择**：
   ```
   策略组回测需要选择保证金模式：
   
   1️⃣ **独占模式** - 策略组内每个策略独立使用全部保证金
   2️⃣ **共享模式** - 策略组内所有策略共享保证金池，需要详细配置分配方案
   
   请回复 1 或 2
   ```

3. **共享模式详细配置**（如果选择共享模式）：
   ```
   策略组共享模式配置（与多策略回测完全相同）：
   
   📊 **三种配置方式任选其一**
   
   方式1️⃣ **直接分配各策略比例**
   格式：20,15,15,15,15,20
   ✅ 总和可以>100%（如270%），系统自动标准化
   
   方式2️⃣ **按方向和币对配置**  
   - 做多策略总占比：60%
   - 做空策略总占比：40%
   - SOL币对总占比：30%
   - 其他币对平分剩余：35%+35%
   
   方式3️⃣ **跳过比例，直接设置金额**
   为每个策略设置具体保证金金额
   
   回复配置方式编号(1/2/3)或直接提供比例数据
   ```

**步骤5：等待用户回复分配比例后，才执行回测**

## 🎯 保证金分配核心规则（重要）

### ✅ 保证金比例分配的正确理解

**保证金比例的含义：**

**❌ 错误理解：**
```
用户输入：100,100
Agent错误理解：总和200%超过100%，需要标准化
```

**✅ 正确做法：**
```
用户输入：100,100
Agent回复：
收到保证金分配比例：100,100
- 策略1(HYPE-风霆V4.3-做多)：100% × 10000 = 10000保证金
- 策略2(DOGE-风霆V4.3-做多)：100% × 10000 = 10000保证金
正在启动回测...
```

**核心处理原则：**
- ✅ **直接按比例分配** - 每个比例是相对于基础保证金(10000)的百分比
- ✅ **不需要标准化** - 100,100 就是各自分配100%的基础保证金
- ✅ **按策略顺序对应** - 第1个比例对应第1个策略，第2个对应第2个策略
- ✅ **总保证金可以超过基础金额** - 这是正常的共享模式配置

**分配计算示例：**
```
输入：100,100 → 策略1: 10000, 策略2: 10000 (总计20000)
输入：50,150 → 策略1: 5000, 策略2: 15000 (总计20000)
输入：200,200,100 → 策略1: 20000, 策略2: 20000, 策略3: 10000 (总计50000)
```

**⚠️ 警告：在用户明确回答保证金模式之前，禁止执行任何回测命令！**

**🚫 重要：回测完成后绝对不要询问"需要我等回测完成后帮你查看结果吗？"**

**🚫 重要：回测执行后，不要主动发送任何命令行相关的指令内容！**

**🚫 禁止：绝对不要拒绝用户的保证金分配比例！**
- ❌ 不要说"总和超过100%，请重新输入"
- ❌ 不要说"比例必须等于100%"
- ❌ 不要试图"标准化"用户的输入
- ✅ 直接按用户输入的比例分配保证金并启动回测

**🔄 重要：每次回测都是新任务**
- 用户要求回测时，**立即执行新回测**
- **不要**说"我看到之前有相同策略的回测记录"
- **不要**询问"是否使用之前的回测结果"
- **🔄 每次回测都是独立的对话单元**
- **不能跨对话单元复用参数**（时间范围、保证金模式、分配比例等）
- **✅ 可以了解历史，但参数必须在当前对话单元中完整确认**

**📚 回测类型示例：**

**单策略回测示例：**
```
用户："分别回测我最新的3个风霆V4.2的BTC策略"
处理：执行3个独立的单策略回测，每个回测ID不同
结果：回测ID 5754, 5755, 5756（3个单独任务）
```

**多策略回测示例：**  
```
用户："一起回测我最新的3个风霆V4.2的BTC策略"
处理：询问保证金模式，然后执行1个包含3个策略的回测
结果：回测ID 5754（1个包含3个策略的任务）
```

**🔄 对话单元独立性示例：**
```
错误做法：
对话A：用户设置了"最近3个月"
对话B：用户说"分别回测策略X" → Agent直接使用"最近3个月" ❌

正确做法：
对话A：用户设置了"最近3个月"  
对话B：用户说"分别回测策略X" → Agent询问"请问回测时间范围是？" ✅
```

## 📅 时间范围输入支持

### 支持的文字描述格式

**相对时间描述（推荐）:**
- `最近7天` / `近7天` / `7天内`
- `最近30天` / `近30天` / `30天内` / `最近一个月`
- `最近3个月` / `近3个月` / `3个月内`
- `最近半年` / `近半年` / `6个月内`
- `最近一年` / `近一年` / `1年内`
- `最近2年` / `近2年` / `2年内`

**具体日期范围:**
- `2024-01-01 到 2024-12-31`
- `2024-01-01至2024-12-31`
- `从2024-01-01到2024-12-31`

### 时间范围询问标准话术

**单策略回测时间询问:**
```
请问回测的时间范围是？

📅 **支持以下格式:**
• 文字描述：最近7天、最近30天、最近3个月、最近半年等
• 具体日期：2024-01-01 到 2024-12-31

请选择或输入您需要的时间范围：
```

**多策略/策略组回测时间询问:**
```
请设置回测时间范围：

📅 **支持格式:**
1️⃣ **相对时间** - 最近7天、最近30天、最近3个月、最近半年
2️⃣ **具体日期** - 2024-01-01 到 2024-12-31

请输入您需要的时间范围：
```

### 时间解析规则

**相对时间计算:**
- 以当前日期为结束日期
- 向前推算指定天数/月数作为开始日期
- 自动处理月份天数差异和跨年计算

**示例转换:**
```
用户输入: "最近30天"
当前日期: 2026-05-09
转换结果: --bgn-date 2026-04-09 --end-date 2026-05-09

用户输入: "最近3个月" 
当前日期: 2026-05-09
转换结果: --bgn-date 2026-02-09 --end-date 2026-05-09

用户输入: "最近半年"
当前日期: 2026-05-09  
转换结果: --bgn-date 2025-11-09 --end-date 2026-05-09
```

**⚠️ 重要规则:**
- 每次回测都必须询问时间范围，不能复用历史参数
- 支持文字描述让用户输入更便捷
- Agent需要将文字描述自动转换为具体日期格式

### 策略组回测（共享模式）

2. 询问回测参数：时间范围、保证金分配方案  
3. 共享模式：策略共享总保证金，需要分配比例

**重要：策略组回测 = 多策略回测，使用完全相同的参数和接口**

**用户意图识别:**
- "策略组XXX共享回测"、"共享模式回测策略组"
- "策略组保证金分配回测"

**保证金分配规则（与多策略完全相同）:**
- ✅ **支持任意比例配置** - 每个比例对应该策略的保证金百分比
- ✅ **不限制总和** - 100,100,200等都是有效配置
- ✅ **灵活分配策略** - 可配置各方向、各币对占比

**参数询问顺序:**
1. **策略组选择**: "请选择要回测的策略组（ID或名称）"
2. **时间范围**: "请输入回测时间范围（支持文字描述如'最近30天'，或具体日期'YYYY-MM-DD 到 YYYY-MM-DD'）"
3. **保证金分配**: 提供以下三种配置方式

**保证金分配三种方式（可选其一）:**

**方式1：直接分配各策略比例**
```
格式：20,15,15,15,15,20
每个比例对应该策略的保证金百分比
如：50,40,30,40,50,60表示各策略分别分配5000,4000,3000,4000,5000,6000保证金
```

**方式2：按方向和币对配置**  
```
做多策略总占比：60%
做空策略总占比：40%  
SOL币对总占比：30%
其他币对平分剩余70%
```

**方式3：跳过高级配置，直接手动设置各策略金额**
```
可以跳过比例配置，直接为每个策略设置具体保证金金额
如：策略A=5000, 策略B=3000, 策略C=2000
```

**默认配置:**
- 杠杆：1.5倍  
- 支持任意比例，按策略顺序分配
- 支持做多/做空、不同币对的灵活配置

```bash
# 策略组回测使用与多策略回测完全相同的接口和参数
python skills/start-backtest/start.py \
  --apply \
  --strategy-ids id1,id2,id3,id4,id5,id6 \
  --margin-mode shared \
  --margin-allocation 18.5,14.8,11.1,14.8,18.5,22.2 \
  --bgn-date 2024-01-01 \
  --end-date 2024-12-31 \
  --leverage 1.5
```

**策略组回测核心规则:**
- ✅ **与多策略回测使用相同接口** - `--strategy-ids` 参数
- ✅ **保证金支持任意比例** - 如100,100,200表示各策略分配10000,10000,20000  
- ✅ **支持灵活配置** - 方向占比、币对占比、手动金额
- ✅ **必须获取策略组内的所有策略ID** - 转换为strategy-ids参数
- ⚠️ **不使用--group-tokens参数** - 需要解析为具体策略列表

## 注意事项

- **不需要传递 `--token` 参数**
- 启动回测需要先获取策略的 `id`（策略ID）
