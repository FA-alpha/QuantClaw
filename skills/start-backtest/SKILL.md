# QuantClaw 回测技能
技能介绍:该技能是用于fourieralpha项目进行项目回测所用的技能,包含:获取usertoken,查询策略信息,查询策略组信息,回测前置条件检测,保证金分配方案计算,执行回测流程,查询回测等功能

 **行为边界 🚨**： 
 - 严格遵守用户原始意图 
 - 不得自行转换为其他任务 
 - 遇到不确定情况必须询问用户 
 - 禁止在用户未明确说明的情况下擅自跳转到其他技能流程 
 - 缺少参数或参数不明确时主动询问用户
 - 禁止展示命令行参数给用户看
 - **回测成功后不要询问用户是否需要等待或查看结果,也不要说可以帮用户监控回测之类的发言**
 - **每次执行回测必须询问时间范围，不使用历史对话信息, 不去取之前回测的上下文中的时间信息**
 - request.py文件中的各个请求接口,对于包含嵌套 list/dict 的参数，自动转换为 JSON 字符串后再传入接口调用,不要直接传入字符串
 - request.py文件已支持cli调用其中的方法
 - 思考时必须使用中文进行思考和表达

#### UserToken 获取规则
1. 使用 jq 工具从 ~/.quantclaw/users.json 获取
2. 必须传入当前机器人的 agentID
3. 获取失败时需要重试机制

```bash
# UserToken 获取示例
USERTOKEN=$(cat ~/.quantclaw/users.json | jq -r --arg agent_id "当前机器人agentID" '.users[] | select(.agentId == $agent_id) | .token')
```

#### UserToken获取流程
1. 先获取当前机器人的 agentID
2. 通过 agentID 查询对应的 UserToken
3. 检查 UserToken 是否为空
   - 如果为空，等待2秒后重试
   - 最大重试次数为5次
4. 重试5次仍失败，通知用户
### UserToken获取与管理补充说明

#### UserToken验证与存储流程
  - 获取到UserToken之后,进行Token存储

    1. Token存储方式:
    - 使用环境变量 `USERTOKEN_{agent_id}` 进行存储
    - 存储格式: `export USERTOKEN_{agent_id}="{UserToken值}"`
    2. Token重新获取机制:
    • 当接口返回Token错误时
    • 优先读取 USERTOKEN_{agent_id} 环境变量,若该环境变量有值,将该值与接口报错时请求参数中的usertoken进行对比,若完全一致,说明环境变量中的Token无效,则需要清理该环境变量,然后根据UserToken 获取规则重新进行获取
    3. 环境变量生命周期管理:
    • 在Agent关闭前自动清理
    4. 安全注意事项:
    • 谨防Token泄露

#### 注意事项
- UserToken 是唯一的用户认证凭证
- 获取到UserToken之后,调用
- 所有接口调用必须使用此 UserToken
- 不要与其他 Token 概念混淆

### 以下部分为独立模块之——执行回测流程
## 执行回测流程:进行回测时,需要严格按照以下内容的第一步,第二步等步骤按顺序进行,仅在回测单策略时,能跳过第四步,其他情况下不能跳过步骤或打乱步骤顺序

  ### 执行回测流程:第一步：确认回测目标
  - 根据用户输入或上下文确定回测目标
  - 处理不同程度的模糊描述
    - 精确到具体策略ID
    - 精确到策略组ID
    - 模糊描述（如"最新的风霆V4策略"）

    #### 判断目标类型
    - 具体策略：已明确策略ID->进入第二步查找策略规则
    - 其他技能传入的参数: 如果传入的是策略id,策略库id,都代表这是策略id ->进入第二步查找策略流程
    - 具体策略组：已明确策略组ID->进入第二步查找策略组规则
    - 其他技能传入的参数: 如果传入的是策略组id,代表这是策略组id ->进入第二步查找策略组规则
    - 模糊描述：需进入下一步策略查找步骤

  ### 执行回测流程:第二步：策略查找
  - 该步骤仅在用户未明确指定策略/策略组时启用
  #### 查找策略组的规则--仅在查找策略组时使用
    
    - 若已有策略组的id,则直接进入执行回测流程的第三步,不需要再进行后续的查找了
    - 策略组查找：在不知道策略组的id的情况下,调用get_strategy_groups方法,若用户指定策略组名称：使用 `search_val`参数对关键词进行搜索,获取满足条件的策略组的列表,让用户选择某个策略组,获取该策略组的id,然后进入执行回测流程的第三步
    - 可选筛选：状态、分页等

  #### 查找策略的规则--仅在查找多策略或单策略时使用
    - 若是单策略回测,且已经给出策略id,则直接进入到执行回测流程的第三步,回测时间范围检测,若是单策略回测或是多策略回测且未给出具体的策略id,则查看后续规则
    - 策略查找：调用get_strategies方法
    - 时间相关筛选：不传 `search_val`,若根据时间查找策略,则根据返回的参数中的create_time参数去确定策略的创建时间
    - 策略名称筛选：传入具体名称,若是搜索关键词,则用`search_val`参数进行搜索,若已经给多个策略的信息,则不能使用search_val参数搜索,而是先search_val参数传空,然后根据根据返回的策略列表中的相关信息去确定用户需要回测的策略,最后根据这些策略的参数去进行下一步
    - 多策略情况下,必须找到所有的策略,且策略信息内有具体数据(不能是空数组之类)才能进行下一步,否则这里直接报错告诉用户,并且需要用户重新提供策略信息
    - 可选筛选：币种、交易类型、状态等

### 执行回测流程:第三步：回测前置条件检测
  该步骤在用户回测多策略或者策略组时必须进行,若用户回测的是单个策略,则跳过执行回测流程的第四步,直接进行执行回测流程的第五步.
  ### 若用户是从查询回测流程的重新回测功能执行的这一步,则不能采用原有回测中一样的参数,若是多策略的回测需要重新询问用户回测时间范围和保证金模式以及各策略中参数分配比例等信息,若是单策略的回测,则需要重新询问用户回测的时间范围.
  #### 回测时间范围检测
   - 若用户没有指定明确的回测时间范围,而是给出模糊的:回测熊市时间段,选择牛市时间范围等内容,则在这里要调用一次搜索市场行情的独立模块,去通过搜索功能获取一下用户指定的时间范围,如果用户指定的时间范围不合法,则需要提示用户重新提供合法的时间范围
   - 确认回测起止日期
   - 验证日期的合法性
    - 开始日期必须早于结束日期
    - 回测时间范围不得超过5年
    - 回测的时间颗粒度为日
    - 不支持未来日期回测
    - **进行回测时，回测开始时间不得早于2020-01-01**

  #### 保证金模式选择
  - 根据策略数量决定保证金模式
    - 单个策略（1个）：默认选择,且只能选择独立保证金模式
    - 多个策略（>1个）：必需要用户确认保证金模式,不确定就无法进行下一步操作
  #### 初始资金选择
  - 默认10000,不会提示用户去更改初始资金
  #### 杠杆倍数选择
  - 单策略回测时.默认杠杆倍数为1,也不会询问用户关于杠杆倍数的事
  - 多策略或者策略组回测时,若是独立保证金模式,默认杠杆倍数为1,但不会主动提示用户去更改杠杆倍数,除非用户明确需求了
  - 多策略或者策略组回测时,若是共享保证金模式,默认杠杆倍数(leverage)为1.5,但不会主动提示用户去更改杠杆倍数,除非用户明确需求了
  ##### 保证金模式详细说明
  1. 独立保证金模式
    - 每个策略独立分配保证金(每个策略默认10000)
    - 策略间资金相互隔离

  2. 共享保证金模式
    - 多个策略共享总保证金(默认10000)
    - 动态分配资金额度
    - 优化资金利用率
    - 需要精细的资金分配策略

### 执行回测流程:第四步：共享保证金模式下的保证金分配方案计算
#### 仅用于多策略和策略组的共享保证金模式保证金分配流程
  该步骤在用户回测多策略或者策略组时必须进行,若用户回测的是单个策略,则跳过执行回测流程的第四步,直接进行执行回测流程的第五步.
##### 保证金分配前置流程
1. 调用 `check_allocation_completeness()` 方法,若要回测的是策略组,则仅传入strategy_group_id参数,参数内容为策略组id,若要回测的是多策略,则仅传入strategy_ids参数,参数内容为多个策略的id,用,分开
   - 分析策略组或者多策略的保证金分配需求
   - 识别需要补充的分配参数
   - 判断参数是否完整
  ##### 可能返回的具体分配参数,括号内为参数的比例分配说明,用于限制用户输入比例的
  - 币种做多分配比例(该值与币种做空分配比例的值相加总计最多为100%,可以不足100%)
  - 币种做空分配比例(该值与币种做多分配比例的值相加总计最多为100%,可以不足100%)
  - AI时间做多分配比例(AI时间做多分配比例最多为100%,可以不足100%)
  - AI时间做空分配比例(AI时间做空分配比例最多为100%,可以不足100%)
  
2. 参数完整性处理
   - 若存在缺失参数(如返回了ai时间做空的参数,而用户只提到了分配币种的资金,或是者用户提到了分配币种的资金,但没有提到分配给哪个币种了,又或者用户提到了分配了某个币种的资金,但没有提到做多还是做空了等等),则需要询问用户以进行参数补全
     * 通过交互方式询问用户
     * 根据用户提供的参数补充缺失的保证金分配参数
     * 返回的分配参数,每一种都要确认全有值之后,才能进行下一步保证金分配计算阶段
   - 若参数完整
     * 进入保证金分配计算阶段
  - - 特殊情况,正常执行回测流程的情况下无视该条:(若该次回测是由查询回测流程发起的,且在查询回测的步骤就已经获取到了每个策略对应的ai_time_id和ai_time_name参数的情况下,则在进行保证金分配计算时,哪怕没有返回AI时间做多分配比例,AI时间做空分配比例,也需要使用ai_time_name来确定AI时间分配的参数名称,后面根据策略的direction参数来拼接多空方向,如ai_time_name是2025年震荡,direction是long,则拼接出来的参数名称为2025年震荡做多,而ai_time_name则对应着它对应的ai_time_id参数,如该策略参数中,ai_time_id=-6,则展示给用户看的是:需要分配2025年震荡做多的比例,假设用户给了50%,那么在调用 `calc_margin()` 方法的时候,传入的long_ai_time_pcts这个数组中就会增加一个元素:ai_time_id:"-6",pct:50 )
3. 保证金分配计算
   - 检验--是否是多策略或者是策略组的回测,若是,则进行下一步
   - 检查是否在当次执行回测流程中调用了 `check_allocation_completeness()` 方法,若没有调用,则需要重新走保证金分配流程下的第一步步骤去调用`check_allocation_completeness()` 方法,然后走保证金分配流程下的第二步步骤去确认参数完整性
   - 调用 `calc_margin()` 方法
   - 根据策略特征和用户输入
   - 计算每个策略的具体保证金额度
   - 请求时有几个默认参数如下:--leverage 1.5 --long-pct 90 --short-pct 20
   - 默认参数可以更改,但是不会主动提示用户去更改默认参数



### 执行回测流程:第五步：执行回测流程
#### 回测任务提交

##### 回测参数准备
1. 整合前四个步骤收集的信息
   - 要回测的所有策略的ID(可以是单个或多个),当是策略组回测的时候,请求apply_backtest方法中的strategy_ids参数需要包含该策略组中所有策略id
   - 回测时间范围
   - 保证金分配方案(共享保证金模式下,需要根据calc_margin()方法返回的各策略保证金值去分配保证金)
   - 保证金模式（独立/共享）

2. 调用 `apply_backtest()` 方法
   - 提交回测任务
   - 传入必要的回测参数

- `strategy_id`/`strategy_ids`: 回测策略
- `bgn_date`/`end_date`: 回测时间范围
- `init_balance`: 初始资金
- `leverage`: 杠杆倍数
- `margin_mode`: 保证金模式
- `margin_allocation`: 保证金分配方案

2. 构造回测请求

4. 获取回测任务ID
5. 准备监控回测状态

- 记录回测任务ID以便后续跟踪
- **进行回测时，回测开始时间不得早于2020-01-01**
- 回测请求提交之后,该次回测的聊天对话就已经是一个独立的单元内容了,下次进行操作的时候,不要主动代入这次单元中提到的各个参数,除非用户明确需求了1

### 查询回测流程:第四步：重新回测
