# QuantClaw 开启交易机器人技能
技能介绍:该技能是用于fourieralpha项目开启交易机器人所用的技能,包含:获取agentid,查询策略信息,查询策略组信息,查询用户绑定的交易所账户列表,查询对应条件的杠杆倍数,查询该交易所账户下的可用资金,开启单策略交易机器人,开启策略组交易机器人等功能

 **行为边界 🚨**： 
 - 严格遵守用户原始意图 
 - 不得自行转换为其他任务 
 - 遇到不确定情况必须询问用户 
 - 禁止在用户未明确说明的情况下擅自跳转到其他技能流程 
 - 缺少参数或参数不明确时主动询问用户
 - 禁止展示命令行参数给用户看
 - **创建交易机器人成功后不需要问用户是否要查看机器人详情**
 - **每次创建机器人前必须先确认是模拟盘还是实盘**
  - **每次用户创建机器人时,不得从上一次创建机器人的历史记录中去获取创建机器人所需的参数,而是应该严格按照用户需求,判断是需要创建单策略机器人还是策略组机器人,然后走对应的独立模块之开启单策略机器人流程或是独立模块之开启策略组机器人流程**
 - request.py文件中的各个请求接口,对于包含嵌套 list/dict 的参数，自动转换为 JSON 字符串后再传入接口调用,不要直接传入字符串
 - request.py文件支持cli调用其中的方法,但是调用时需要注意:若使用cli方法,则Typer会自动将 Python 函数名中的下划线转换为连字符：例如，函数名 `get_strategy_with_id` 将对应 CLI 命令 `get-strategy-with-id`，在使用 CLI 调用时需要使用连字符而不是下划线。
 - request.py文件进行接口请求时,若返回了"strangeError"的文字,则直接停止当前操作,然后告诉用户当前在哪个步骤遇到了问题,当次思考流程绝对不能在接口返回strangeError之后还继续做任何网络请求,除非用户发送了下一条对话且明确允许进行操作.
### 🔑 重要: agentID使用方法
#### agentID 获取规则
1. 获取当前机器人的agentID
2. 调用request.py文件的接口时,若接口需要传递agent_id参数,则将当前机器人的agentID传入,数据类型为string,request.py中会自动去获取usertoken


### 本skill分为多个独立模块,模块之间可能会在走到某个步骤之后去调用其他模块,但每个模块的功能互相独立,互不干扰,运行模块时,所有功能限定在模块的说明范围之内,不要自己去主动创建脚本之类的

### 以下部分为独立模块之——开启单策略交易机器人流程
  ## 开启单策略交易机器人流程:开启交易机器人时,若用户仅选择了一个策略,需要严格按照以下内容的第一步,第二步等步骤按顺序进行,不能跳过步骤或打乱步骤顺序

  ### 开启单策略交易机器人流程:第一步：确认机器人的策略
  - 根据用户输入或上下文确定该机器人要使用的策略,若传入的是策略id(一串纯数字),则调用request.py中的get_strategy_with_id方法(该方法必须传入strategy_id参数和agent_id参数)去根据策略id获取该策略的详细信息(如策略参数等),若传入的不是纯数字,则通过调用request.py中的get_strategy_lists方法,根据返回的策略列表信息,去匹配用户输入的策略信息,找到用户需要用来开启机器人的策略,获取该策略的id以及关键参数,其中策略中的create_time参数来判断策略创建时间,然后进入开启单策略交易机器人流程的第二步
    - 若要查询到匹配策略列表中的策略,需要通过调用request.py中的get_strategy_lists方法,根据返回的策略列表信息,去匹配用户输入的策略信息,找到用户需要用来开启机器人的策略,获取该策略的id以及关键参数,然后进入开启单策略交易机器人流程的第二步
  ### 开启单策略交易机器人流程:第二步：确认机器人的基础信息
  - 获取用户绑定的交易所账户列表,然后通过询问用户的方式,确认用户开启机器人时要用到的交易所账户,若用户没有确认,则不往下进行
    - 查询交易所账户,通过调用request.py中的get_exchange_lists方法,获取用户绑定的交易所账户列表,并展示给用户看,让用户选择要用来开启机器人的交易所账户
  - 通过询问用户的方式,确认用户要开启的是模拟盘还是实盘,若用户没有确认,则不往下进行
  - 通过询问用户的方式,确认用户要开启的机器人的保证金模式,是逐仓模式还是全仓模式,若用户没有确认,则不往下进行.
  - 通过询问用户的方式,确认用户要开启的机器人的币对模式,是币本位还是usdt本位还是usdc本位,若用户没有确认,则不往下进行.
  - 通过调用request.py中的方法,检查该币对(根据策略详细信息可以确定该策略的交易币种,根据币对模式确定交易模式,根据选择的交易所账户确定交易所)在对应的交易所中是否存在,若存在,则进行下一步,若不存在,则让用户重新选择币对模式
  - 在确认了机器人的币对模式之后,通过调用request.py中的get_balance方法来获取用户在该交易所的当前策略所对应的币对名称,目前已有的,该交易所下的该币对杠杆是几倍,用户的可用资金和理财账户资金有多少
    - get_balance方法参数如下:
      - "usertoken":用户登录token
      - "account_id":交易所账户ID
      - "basic_unit":币本位/U本位（USD-币本位 USDT-usdt本位 USDC-usdc本位）
      - "coin": 币种
      - "strategy_id": 策略ID，用于判断是现货还是合约
  - 在获取了get_balance方法返回的参数之后,如果用户要开启的机器人是实盘机器人,则根据返回的余额和理财余额,后续用户在设置下单金额或者投资总金额的时候,可以提醒用户量力而行,给出推荐的下单金额,如果用户要开启的机器人是模拟盘机器人,则不需要提示用户,因为模拟盘不涉及真实资金,用户可以随意设置下单金额或者投资总金额.
    - get_balance方法返回值如下:
      - "balance": 余额
      - "finance_balance": 理财余额
      - "unit": 单位（USDT/ETH/BTC等）
      - "multiple_num":杠杆倍数
  - 自动赎回理财参数默认为自动赎回理财,不会主动询问用户,除非用户主动要求修改
  ### 开启单策略交易机器人流程:第三步: 根据策略类型,询问用户创建机器人所需要的参数
  - 开启单策略交易机器人的时候,在询问了用户第二步的所有参数之后,还需要询问用户:
    - type=1,2,11 的时候,需要询问用户确认:"fst_capital":初次下单金额,"each_capital"加仓下单金额,"max_grid_size":最大加仓次数,"total_investment":总计投资金额,然后根据策略中的add_amt_multiples参数确认加仓金额倍数参数,当用户给出其中一部分参数时,通过调用request.py中的calculate_investment_smart方法去计算剩下的参数的数值,调用规则如下:当该策略的add_amt_multiples参数为1的时候,用户可以设置初次下单金额,加仓下单金额和最大加仓次数,或者只设置投资金额和最大加仓次数,剩下的参数我们用calculate_investment_smart方法帮他计算,当该策略的add_amt_multiples参数不为1的时候,用户不能设置投资金额,只能设置加仓下单金额和最大加仓次数
    - 若用户修改了策略里原有的参数,如投资金额,加仓次数等,需要先把修改后的所有要用来开启机器人的信息都给用户看一遍,用户确认无误了,才能进行下一步操作
    
  ### 开启单策略交易机器人流程:第四步：根据策略类型确认机器人需要的必备的额外参数
  - 根据用户用来开启机器人的策略中的参数type,用户在创建机器人的时候需要一些必备的额外的参数,规则如下:
    - type=1,2(风霆现货/合约)时需要额外传:1:"fst_capital": 初次下单金额   2:"each_capital": 加仓下单金额 3:"max_grid_size": 最大加仓次数 
    - type=3时需要额外传"initial_capital": 投资总金额 
    - type=4时需要额外传 1:"initial_capital": 投资总金额  2:"trade_buy_type": 买入类型（{market}代表吃单 {limit}代表挂单）
    - type=5时需要额外传 1:"initial_capital": 投资总金额  2:"each_capital": 单笔买入/卖出金额 
    - type=8时需要额外传 1:"trade_model": 交易类型（{all}两者全选 {long}做多 {short}做空） 2:"trade_buy_type": 买入类型（{market}代表吃单 {limit}代表挂单）
    - type=11时需要额外传:1:"fst_capital": 初次下单金额   2:"each_capital": 加仓下单金额 3:"max_grid_size": 最大加仓次数 
    - type=25时需要额外传"initial_capital": 投资总金额
    - type=28时需要额外传"initial_capital": 投资总金额
  - backtest_date这个参数在用户选择的策略中,策略列表返回的参数里,若该策略属性中的is_backtest_single=1的话,这个参数必需要传入,他的默认值是当年的1月1日,也可以通过询问用户,让用户确认来得知具体的时间,传入格式是:YYYY-MM-DD
  ### 开启单策略交易机器人流程:第五步：确认无误后创建交易机器人
  - 将前面步骤中确认的必备的参数:
  - 以及根据策略类型确认的额外参数,传入开启单策略交易机器人的接口,创建交易机器人
    - 调用request.py中的apply_trade_bot方法,将前面步骤中确认的必备的参数,以及根据策略类型确认的额外参数,传入开启单策略交易机器人的接口,创建交易机器人,必填参数部分说明如下:
    -   :strategy_id: 开启机器人用到的策略ID
        :name: 机器人名称
        :account_id: 交易所账户ID
        :trade_type: 交易类型（1-模拟盘 2-实盘）
        :multiple_num: 杠杆倍数
        :basic_unit: 币本位/U本位（USD-币本位 USDT-usdt本位 USDC-usdc本位）
        :backtest_date: 回测信号开启日期
        :auto_redeem: 资金不足时是否自动赎回理财,true是赎回,false是不赎回
        :margin_mode: 保证金模式,是逐仓还是全仓,cross是全仓,isolated是逐仓

### 以下部分为独立模块之——开启策略组交易机器人流程
  ## 开启策略组交易机器人流程:开启交易机器人时,若用户选择了一个策略组,需要严格按照以下内容的第一步,第二步等步骤按顺序进行,不能跳过步骤或打乱步骤顺序

  ### 开启策略组交易机器人流程:第一步：确认机器人的策略组信息
  - 根据用户输入或上下文确定该机器人要使用的策略组,如用户尚未指定策略组,则通过get_strategy_groups方法进行查询,然后展示给用户让用户进行选择,只能选择一个策略组
  - 若用户已指定并且确定了策略组id,通过get_strategy_group_with_groupid方法去根据策略组id获取该策略组下全部策略的信息(如策略id,策略参数等),让用户选择是其中所有策略都要开启机器人,还是仅策略组中的部分策略开启机器人

  ### 开启策略组交易机器人流程:第二步：确认该次投资的基础信息
  - 查询用户当前绑定的交易所账户列表,然后通过询问用户的方式,确认用户开启机器人时要用到的交易所账户,若用户没有确认选择哪个交易所账号,则不往下执行流程,直到用户确认了交易所账户为止.
    - 查询交易所账户,通过调用request.py中的get_exchange_lists方法,获取用户绑定的交易所账户列表,并展示给用户看,让用户选择要用来开启机器人的交易所账户
  - 通过询问用户的方式,确认用户要开启的是模拟盘还是实盘,若用户没有确认,则不往下进行
  - 通过询问用户的方式,确认用户要开启的机器人的保证金模式,是逐仓模式还是全仓模式,若用户没有确认保证金模式是逐仓还是全仓,则不往下执行流程,直到用户确认了保证金模式为止.
  - 通过询问用户的方式,确认用户要开启的机器人的币对模式,是币本位还是usdt本位还是usdc本位,若用户没有确认币对模式,则不往下执行流程,直到用户确认了币对模式为止.
  - 通过询问用户的方式,确认用户要开启的策略组机器人的杠杆倍数(所有要开启的机器人的杠杆倍数统一),默认有1倍杠杆,1.5倍杠杆,3倍杠杆3种选项供用户选择,若用户选择大于3倍的杠杆倍数,则提示用户:该杠杆倍数较高,您的投资风险较高,是否确认要开启? 若用户没有确认杠杆倍数的数值,则不往下执行流程,直到用户确认了杠杆倍数的数值为止.

  ### 开启策略组交易机器人流程:第三步：确认该次投资的保证金配置方案
  - 用户需要配置策略组中,做多/做空的保证金占比,多空保证金占比的作用为:假设总计保证金为10000,若做多保证金占比为90%,做空保证金为20%,则等于实际总投资为11000(110%),做多保证金和做空保证金的上限均为100%,也就是最高的实际总投资可以配置为20000(200%),但是默认情况下,默认配置为做多保证金占比为90%,做空保证金占比为20%,这个配置可以不用用户确认就走到下一步,但是必须告知用户,已默认将做多保证金配置为90%,做空保证金配置为20%
  - 做多/做空各币种保证金占比,币种保证金的占比作用为:做多方向的币种种类为该策略组内,所有做多的币种的保证金占比,其值为之前一条的,用户设置的做多的保证金占比,做空方向的币种种类为该策略组内,所有做空的币种的保证金占比,其值为之前一条的,用户设置的做空的保证金占比.假设一个策略有3个做空币种,2个做多币种,且前一步默认的做多保证金占比为90%,做空保证金占比为20%,则这三个做空币种能分配的总计的保证金金额为20%,每个做空币种的保证金分配都必须大于0%小于20%,且其加和为20%,做多也是同理,这两个做多币种能分配的踪迹的保证金金额为90%,每个做多币种的保证金分配都必须大于0%小于90%.
  - 进行类型占比分配,调用request.py的接口,先获取策略组中的各市场行情类型(然后分配各市场行情保证金占比,且确保各市场行情保证金占比相加刚好等于100%),
  ### 开启策略组交易机器人流程:第四步: 告知用户该次机器人开启,各机器人所分配到的保证金金额
  - 根据之前步骤所获取的参数,去请求request.py中的接口,获取各个策略所分配到的保证金金额,然后告知用户如下信息:
    - 交易账户名称,方案币种,保证金模式,投资总金额,交易所杠杆比例,保证金杠杆比例,做多保证金占比,做空保证金占比,每个市场行情类型中,所分配到的保证金占比,要开启的机器人的数量,总保证金金额,每个策略的名称,方向,市场类型和该策略分配到的保证金
  ### 开启策略组交易机器人流程:第五步: 执行开启策略组交易机器人的行为
  - 根据之前步骤所获取的参数,去请求request.py中的接口,开启策略组交易机器人
